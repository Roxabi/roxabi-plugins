## Context

GitHub issue #20 identifies 8 structural pain points across the plugin codebase:

- **Untyped data** — raw dicts cross boundaries in vault, cv, web-intel, and get-invoice-details (linkedin-apply is the exception with dataclasses)
- **No repository pattern** — plugins depend on VaultDB directly or use raw sqlite3 (get-invoice-details bypasses VaultDB entirely)
- **Hardcoded dispatch** — web-intel's scraper.py uses an if/elif chain to route URLs to fetchers
- **Generic exceptions** — `Exception` raised/caught broadly; only linkedin-apply has a custom hierarchy
- **Duplicated infrastructure** — `_lib/paths.py` vendored 5× (byte-identical)
- **Mixed access patterns** — dev-core's github.ts mixes REST, GraphQL, and CLI with no unified port
- **No config validation** — JSON configs loaded as raw dicts, malformed input causes silent failures
- **No use case layer** — business logic scattered across CLI entrypoints

The approved [analysis](../analyses/20-clean-architecture-analysis.mdx) evaluated three migration shapes and recommended **Bottom-Up Foundation**: domain models and port interfaces first (pure types, zero behavior change), then adapter wrapping plugin by plugin, then use case extraction. This was chosen over Plugin-by-Plugin Vertical Slices (pattern drift risk across 6+ plugins) and Extract SDK First (premature port design before migration experience).

**Affected plugins:** vault, web-intel, linkedin-apply, cv, get-invoice-details, image-prompt-generator (Python) + dev-core (TypeScript).

## Goal

Establish Clean Architecture boundaries (domain → ports → adapters) across all plugins so that infrastructure concerns (storage, HTTP, CLI) are isolated behind port interfaces, enabling independent testing, swappable backends, and typed domain objects throughout.

## Users

- **Primary:** Plugin developers (Roxabi team) — every new plugin or feature benefits from clear boundaries and typed models.
- **Secondary:** Contributors and consumers of individual plugins — they get config validation, specific error types, and decoupled infrastructure.

## Out of Scope

- **SDK extraction (Phase 5)** — `_lib/paths.py` deduplication and a shared `roxabi-sdk` package are deferred to a future issue. The vendoring pattern continues for now. This means analysis outcome 5 ("a change to shared path resolution logic requires exactly one edit") is NOT addressed by this spec.
- **Database migration** — this refactor creates the abstraction layer; actual SQLite → PostgreSQL migration is separate.
- **New plugin features** — purely structural refactoring.
- **DI container** — dependency injection is manual (constructor injection at composition roots). No DI framework.
- **get-invoice-details vault access migration** — this plugin uses raw sqlite3 against vault.db from a separate plugin. Migrating it to EntryRepository requires either vendoring the adapter cross-plugin or the SDK. Deferred to Phase 5. **This is a known exception, not a pattern** — no new direct sqlite3 writers should be added.
- **Legacy shim removal** — shims (VaultDB, github.ts wrappers) remain importable throughout this spec. Removal is a follow-up after all consumers have migrated and the SDK is in place. Shims must carry `@deprecated` JSDoc (TS) or docstring (Python) from creation. For TypeScript shims, a biome lint rule must be added to `biome.json` that flags imports of `skills/shared/github.ts`, `skills/shared/config.ts`, and `skills/shared/workspace.ts` from any file outside the shims themselves.

## Prerequisites

Before the first adapter slice (Slice 2) merges, the following infrastructure must be in place:

1. **Pytest in CI** — add a `pytest` step to `.github/workflows/ci.yml` covering `plugins/vault`, `plugins/linkedin-apply`, `plugins/web-intel`, `plugins/cv`, and `plugins/image-prompt-generator`. Each Python plugin with tests must add `pytest` to its `pyproject.toml` dev dependencies.
2. **Python package structure** — all new `domain/`, `ports/`, `adapters/` directories must include `__init__.py` files. Entrypoint `sys.path` manipulation must be updated so sibling packages are importable.
3. **Phase 5 tracking issue** — a GitHub issue must be opened for SDK extraction and shim removal before Slice 7 merges, to prevent indefinite deferral.

## Expected Behavior

### Phase 1+2: Foundation (domain + ports)

A developer opens any plugin and finds two new directories:
- `domain/` — dataclass models (Python) or typed interfaces (TypeScript) representing the plugin's core data. Frozen, validated, self-documenting.
- `ports/` — abstract interfaces defining what the plugin needs from infrastructure (storage, fetching, config loading) without specifying how.

No runtime behavior changes. All existing code continues to work. The new types and interfaces are importable but not yet wired.

### Phase 3: Adapter wrapping (per plugin)

For each plugin in priority order (vault → linkedin-apply → dev-core → web-intel → cv → image-prompt):

An `adapters/` directory appears containing concrete implementations of each port. Existing infrastructure code (VaultDB, BaseFetcher, github.ts functions) is wrapped inside adapters — not rewritten. Consumers switch from importing infrastructure directly to importing port interfaces. Construction happens at the entrypoint (composition root).

**Python example (vault):** A developer adding vault storage to a new plugin depends on `EntryRepository` (port), not `VaultDB` (adapter). The CLI entrypoint (`manage_vault.py`) constructs `SqliteEntryRepository(db_path)` and passes it to business logic. All raw `db.conn.execute()` calls in `manage_vault.py` must be replaced with adapter calls — not kept alongside the adapter. `content_indexer.py` also uses VaultDB directly and must be migrated in the same slice.

**Python example (web-intel):** A developer adding a new fetcher creates a class implementing `FetcherPort` (a thin interface: `supports(url) → bool`, `fetch(url) → FetchResult`), adds it to the `FetcherRegistry`, and is done — no scraper.py modification needed. `BaseFetcher` remains as an internal base class handling caching and CLI concerns.

**TypeScript example (dev-core):** A developer working on a new skill that needs GitHub data imports `IssuePort` or `ProjectPort` (role interfaces) instead of calling `ghGraphQL()` or `run(['gh', ...])` directly. The skill's entry file constructs the adapter and passes it in. The dashboard entry (`dashboard.ts`) serves as the composition root — it constructs `GitHubAdapter`, `EnvConfigAdapter`, etc. and passes them to the rendering/fetching logic.

### Phase 4: Use case extraction

CLI entrypoints (`manage_vault.py`, `scraper.py`, `generate_cv.py`) become thin shells:
1. Parse arguments
2. Construct dependencies (adapters)
3. Call a `UseCase` class that contains the business logic
4. Format and print output

Business logic is now unit-testable without I/O.

## Breadboard

### Python plugin structure (per plugin, after migration)

```
plugins/<plugin>/
├── domain/
│   ├── models.py          # @dataclass(frozen=True) domain objects
│   └── exceptions.py      # Plugin-specific exception hierarchy
├── ports/
│   ├── repository.py      # EntryRepository ABC (vault consumers)
│   ├── fetcher.py          # FetcherPort ABC (web-intel)
│   └── config.py           # ConfigLoader ABC with validation
├── adapters/
│   ├── sqlite_repository.py  # SqliteEntryRepository(EntryRepository)
│   ├── fetcher_registry.py   # FetcherRegistry (replaces if/elif)
│   └── json_config.py        # JsonConfigLoader(ConfigLoader)
├── use_cases/
```

Not every plugin needs every directory — only the relevant ports:

| Plugin | domain/ | ports/ | adapters/ |
|--------|---------|--------|-----------|

| web-intel | FetchResult, URL | FetcherPort (new thin interface) | BaseFetcher subclasses (internal) + FetcherRegistry |
| linkedin-apply | LinkedInJob (exists), MatchResult (exists), ApplicationRecap (exists) | StoragePort (new ABC: `save_analysis`, `load_recap`, `list_applications`, `get_daily_count`, `get_stats`, `find_existing_analysis`), MatcherPort | FileStorageAdapter (new class wrapping storage.py functions; accepts `base_dir` constructor arg for test isolation), ClaudeMatcherAdapter. ### FetcherPort design decision

`FetcherPort` is a **new thin interface**, not a rename of `BaseFetcher`:
```python
class FetcherPort(ABC):
    @abstractmethod
    def supports(self, url: str) -> bool: ...
    @abstractmethod
    def fetch(self, url: str) -> FetchResult: ...
```

`BaseFetcher` remains as an internal implementation base class that all concrete fetchers inherit from. It keeps the caching logic (`cached_fetch`, `get_cache`), URL validation, and CLI support — these are infrastructure concerns that belong inside the adapter layer, not in the port interface.

### StoragePort design note (linkedin-apply)

`storage.py` currently has module-level functions (`save_analysis`, `load_recap`, `list_applications`, `get_daily_count`, `get_stats`, `find_existing_analysis`), not a class. Module-level globals (`APPLICATIONS_DIR`, `INDEX_FILE`) are evaluated at import time, coupling the module to `paths.py` at class-definition time. Migration requires:
1. Define `StoragePort(ABC)` with **all** function signatures as abstract methods (including `get_daily_count`, `get_stats`, `find_existing_analysis`)
2. Create `FileStorageAdapter` class that accepts `base_dir` as a constructor argument (not a module-level global) — enables test isolation without hitting `~/.roxabi-vault`
3. Move current function bodies into the class, replacing module-level path globals with `self.base_dir`
4. Update callers to use the class instance

### GitHubPort design decision: role interfaces

`GitHubPort` is split into **role interfaces** rather than one wide interface:
- `IssuePort` — issue CRUD, labels, comments (REST-backed)
- `ProjectPort` — project fields, items, sub-issues, workflows (GraphQL-backed, some CLI)

`GitHubAdapter` implements both. Skills import only the port they need. This avoids a 15+ method monolithic interface and makes dependencies explicit per skill.

`WorkspacePort` is intentionally separate — it wraps git/worktree operations, not GitHub API calls.

```
       └── registry.get_fetcher(url) → first where supports(url)==True

```

```
       └── implements IssuePort + ProjectPort
  └── constructs EnvConfigAdapter()
       └── implements ConfigPort
```

```
       └── projectPort.updateField(itemId, fieldId, value)
```

## Implementation Constraints

These constraints apply across all slices and must be followed during implementation.

### Security

1. **FTS5 sanitization** — `Fts5SearchAdapter.search()` must delegate to (or identically reproduce) the double-quote escaping in `fts.py:search_entries()` (`query.replace('"', '""')`). Add a unit test asserting correct behavior with FTS5 operators (`OR`, `AND`, `*`, `"`, `NEAR`).
2. **Token storage** — `GitHubAdapter` must store the GitHub token as a private field (`#token` or `private readonly`). The token must never be forwarded to domain objects, exception messages, or serializable properties.
3. **Exception scrubbing** — custom exception classes must not embed raw HTTP response bodies or adapter-internal state in `.message`. Adapters catch raw errors and rethrow as domain-safe exceptions with sanitized summaries (e.g., HTTP status code only). Full response bodies may be logged at debug level only.
4. **Adapters must not import from shims** — shims delegate to adapters, never the reverse. This prevents circular imports and preserves the dependency rule.

### Construction & Initialization

5. **Lazy token resolution** — `GitHubAdapter` must not call `getToken()` or read `process.env.GITHUB_TOKEN` at construction time. Token resolution must be deferred to first use. This prevents vitest from throwing when the adapter module is imported in tests.
6. **No module-level filesystem side-effects** — Python adapters must not evaluate paths (e.g., `get_plugin_data()`) at module scope. All path resolution must happen in the constructor or methods, accepting `base_dir`/`db_path` as constructor arguments. This enables test isolation with `tmp_path` fixtures.
7. **Shim eagerness** — `github.ts` (shim) currently imports `GH_PROJECT_ID` and `GITHUB_REPO` from `config.ts` at the top of the file, triggering eager env reads. After migration, the shim must defer these reads to function call time.

### Backward Compatibility

8. **`FetchResult` JSON stability** — if `FetchResult.to_dict()` changes JSON key names from the current raw dict format, all skills calling `python scraper.py <url>` break. The dataclass field names must match the existing dict keys, or a `to_legacy_dict()` method must be provided.
9. **Redirect orchestration** — the redirect logic that wraps the if/elif dispatch in `scraper.py` is use-case logic. It has no declared home between Slice 5 (registry) and Slice 7 (use cases). During Slice 5, keep it in `scraper.py` as-is; extract to a UseCase in Slice 7.
