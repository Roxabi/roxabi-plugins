## Context

Promoted from `artifacts/analyses/425-ptc-evaluation.mdx` (approved 2026-02-28).

PTC (`code_execution_20260120`) lets Claude emit a Python code block that orchestrates multiple tool invocations inside a sandboxed container. Only the final output returns to the conversation — intermediate file reads, grep results, and computed structures stay in Python variables and never inflate the context window.

Anthropic measured 37% token reduction on complex multi-tool tasks and 85% savings on tool definitions via deferred loading. Dev-core analysis-heavy skills (`/analyze`, `/review`, `/spec`, `security-auditor`) currently issue 20+ sequential Grep/Read calls per session — a prime candidate for this optimization.

## Goal

Determine whether PTC delivers meaningful token savings in dev-core analysis-heavy skills, identify the constraints and gotchas, and produce a written adoption decision (adopt broadly / adopt selectively / defer).

## Users

- **Primary:** dev-core plugin maintainers — consuming the decision document to decide whether to update skill implementations.
- **Secondary:** end users of dev-core skills — benefit indirectly if adopted (faster analysis, longer useful sessions).

## Expected Behavior

1. **POC implementation:** A new experimental agent file `security-auditor-ptc.md` is created alongside the production `security-auditor.md`. It replaces the import graph resolution phase with a single PTC Python code block that traverses transitive imports. The production agent is not modified.

2. **Baseline measurement:** Run the current `security-auditor` (sequential Grep) on `roxabi-boilerplate` as the fixed target project. Count tool calls the agent emits during the import-graph phase and record wall-clock latency (via Bash `time` wrapping). Repeat for ≥ 3 runs. Note: input token counts are not directly observable from within a skill run; tool call count is the primary measurable signal for context inflation.

3. **PTC measurement:** Run `security-auditor-ptc.md` on the same target (`roxabi-boilerplate`). Record the same metrics for ≥ 3 runs. Compute mean ± range for tool calls and latency_ms, and the reduction percentage for each.

4. **ZDR posture:** Before writing the decision, document the assumed ZDR user fraction for dev-core's target user base (e.g. `"unknown"`, `"<10%"`, `">50%"`). This assumption gates the recommendation scope: if ZDR fraction is high, PTC may be unsuitable regardless of savings. The decision document is the gate — no implementation proceeds without it.

5. **Limitation audit:** For each known constraint, document its impact based on published API behavior (no live test required for ZDR or MCP): ZDR incompatibility (PTC unavailable for ZDR users — document coverage gap), MCP tools unavailable inside PTC (document which dev-core tool calls would be excluded), Python-only runtime (document language constraint), container spin-up latency (estimate from API timestamps in measured runs).

6. **Decision document:** Write `artifacts/analyses/18-ptc-decision.mdx` with findings, per-skill applicability matrix, confidence level, ZDR posture assumption, limitation impact per constraint, and a final recommendation (adopt broadly / selectively / defer + rationale grounded in ZDR posture and limitation findings).
