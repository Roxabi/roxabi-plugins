## Context

Promoted from `artifacts/analyses/425-ptc-evaluation.mdx` (approved 2026-02-28).

PTC (`code_execution_20260120`) lets Claude emit a Python code block that orchestrates multiple tool invocations inside a sandboxed container. Only the final output returns to the conversation — intermediate file reads, grep results, and computed structures stay in Python variables and never inflate the context window.

Anthropic measured 37% token reduction on complex multi-tool tasks and 85% savings on tool definitions via deferred loading. Dev-core analysis-heavy skills (`/analyze`, `/review`, `/spec`, `security-auditor`) currently issue 20+ sequential Grep/Read calls per session — a prime candidate for this optimization.

## Goal

Determine whether PTC delivers meaningful token savings in dev-core analysis-heavy skills, identify the constraints and gotchas, and produce a written adoption decision (adopt broadly / adopt selectively / defer).

## Users

- **Primary:** dev-core plugin maintainers — consuming the decision document to decide whether to update skill implementations.
- **Secondary:** end users of dev-core skills — benefit indirectly if adopted (faster analysis, longer useful sessions).

## Expected Behavior

1. **POC implementation:** A new experimental agent file `security-auditor-ptc.md` is created alongside the production `security-auditor.md`. It replaces the import graph resolution phase with a single PTC Python code block that traverses transitive imports. The production agent is not modified.

2. **Baseline measurement:** Run the current `security-auditor` (sequential Grep) on a representative target project (e.g. `roxabi-boilerplate`). Record input token count, tool call count, and wall-clock latency for the import-graph phase. Repeat for ≥ 3 runs.

3. **PTC measurement:** Run `security-auditor-ptc.md` on the same target. Record the same metrics for ≥ 3 runs. Compute mean ± range for each metric and reduction percentage.

4. **ZDR posture:** Before writing the decision, document the assumed ZDR user fraction for dev-core's target user base (e.g. `"unknown"`, `"<10%"`, `">50%"`). This assumption gates the recommendation scope.

5. **Limitation audit:** Exercise known constraints — ZDR incompatibility, MCP tools unavailable inside PTC, Python-only runtime, container spin-up latency — and document how each affects adoption scope.

6. **Decision document:** Write `artifacts/analyses/18-ptc-decision.mdx` with findings, per-skill applicability matrix, confidence level, ZDR posture assumption, and a final recommendation (adopt broadly / selectively / defer + rationale).
