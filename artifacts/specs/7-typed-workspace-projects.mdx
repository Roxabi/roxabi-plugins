## Context

Promoted from `artifacts/analyses/typed-workspace-projects.md` (2026-03-03, Draft).
Frame: `artifacts/frames/7-typed-workspace-projects.mdx` (approved).

The dashboard hardcodes field IDs and column rendering for a single "technical" project schema.
Ryvo needs to display a **tech issues board** (Status / Size / Priority, CI/Vercel, /dev links)
alongside a **company roadmap board** (Status / Quarter / Pillar, no CI/Vercel, no /dev links)
in the same workspace. This spec defines the typed extension.

## Goal

Extend `WorkspaceProject` with `type` and `fieldIds` so that column headers, field dispatch,
CI/Vercel visibility, and /dev link visibility are all driven per-project ‚Äî with zero breaking
change for existing configs.

## Users

- **Primary:** Engineers configuring a multi-project dashboard (e.g., Ryvo: 1 tech board + 1 company roadmap).
- **Secondary:** Any future team adding a non-technical GitHub Project to their workspace.

## Expected Behavior

### Adding a company roadmap project

An engineer adds a second project to `workspace.json` with `type: 'company'` and runs `/init`
(or manually sets `fieldIds`). The dashboard tab for that project shows:
- Column 2 header: **Quarter** with üìÖ icon
- Column 3 header: **Pillar** with üèõ icon
- No CI/Vercel section
- No `/dev #N` links
- Default sort: Quarter ascending

The tech board tab is unchanged ‚Äî all existing behaviour preserved.

### Inline edits on a company project

Company projects **support inline field edits in v1**. When the user edits a field on a
company project item, `update.ts` uses `project.fieldIds.col2` / `col3` (not the global
`.env` SIZE_FIELD_ID). Dropdown values resolve from `fieldIds.col2Options` / `col3Options`
when present. If a slot's options are absent, that field renders as read-only (no special
guard needed ‚Äî the existing options-check path handles it naturally).

### fieldIds is the complete source when present

When `fieldIds` is present on a `WorkspaceProject`, it is the **complete source** for that
project's field IDs ‚Äî no field-by-field mixing with `.env`. If `fieldIds` is present but
missing a required slot (e.g., `status`), `resolveFieldIds` throws a descriptive config
error at startup (e.g., `"[project ryvo-roadmap] fieldIds.status is required"`), surfacing
misconfiguration early rather than producing a silent runtime failure.

### /init scaffolding

`/init create-project --type company` scaffolds the workspace entry with `type: 'company'`
and queries GitHub to write `fieldIds` (status, col2, col3, options) to `workspace.json`.
The `--type` flag defaults to `'technical'` if omitted. If GitHub returns no matching fields
for a required slot, `/init` warns and writes `fieldIds: {}` for that project rather than
aborting ‚Äî the user can manually populate the missing field IDs.

### Backwards compatibility

Existing `workspace.json` files with no `type` or `fieldIds` continue to work:
- `type` absent ‚Üí treated as `'technical'`
- `fieldIds` absent ‚Üí falls back to `.env` STATUS_FIELD_ID / SIZE_FIELD_ID / PRIORITY_FIELD_ID

## Out of Scope

- Auto-detecting project type by inspecting which GitHub fields exist
- More than two `ProjectType` values in this iteration (`hiring`, `legal`, etc.)
- Roadmap timeline/Gantt view for company type
- Migrating existing `.env` field IDs to `workspace.json` automatically
- Moving secrets/tokens from `.env` (only field IDs move to workspace.json)
