## Goal

Replace the single-project `.env`-based config with a globally installable `roxabi` CLI
and a `workspace.json` file, enabling the dashboard and triage commands to operate across
all registered GitHub Projects in one shot.

## Users

- **Primary:** Mickael — owns `mickaelV0` personal repos + `Roxabi` org, single GitHub
  token, uses dev-core daily across multiple repos.
- **Secondary:** Any dev-core user with more than one GitHub Project board, including
  fresh installs where neither `~/.roxabi-vault/` nor `~/.config/roxabi/` exist yet.

## Expected Behavior

### Installing and first use

After `bun install -g roxabi-cli`, the developer runs `roxabi workspace add Roxabi/roxabi-plugins`.
The CLI queries `repository.projectsV2` via GraphQL. If one project is found, it is written
directly. If multiple projects are found, the user is prompted to choose from a numbered
list (same pattern as `/init` Phase 3). The chosen entry is written to
`~/.roxabi-vault/workspace.json` (vault present) or `~/.config/roxabi/workspace.json`
(fallback). The parent directory is created with mode `0700` if absent.

### Viewing issues across all projects

`roxabi issues` reads `workspace.json`, builds a single batched GraphQL query with one
aliased `node(id: "PVT_...")` per registered project, fires **one HTTP request**, and
renders a combined table. Each row includes the project label as a column.
The response shape is `data.project0`, `data.project1`, … keyed by index — `lib/fetch.ts`
is refactored to accept `WorkspaceProject[]` and return a per-project items map.

### Config layer architecture

`shared/config.ts` is **not rewritten** — it stays env-var-only. A new module
`lib/workspace.ts` reads `workspace.json` synchronously at startup
(`readFileSync` + `JSON.parse`). CLI commands and the dashboard consume `lib/workspace.ts`.
`issues/lib/config.ts` gains a `resolveConfig()` function that checks `lib/workspace.ts`
first, then falls back to `shared/config.ts` env vars — backward compat is preserved at
this boundary.

### Dashboard

Opening the dashboard shows a tab bar: `[ All ] [ Roxabi/repo-a ] [ mickaelV0/repo-b ]`.
The All tab groups issues by project with a section header per project. Each project tab
shows today's single-project view.

The dashboard uses the **existing SSE mechanism** (`/api/events` stream). The server-side
poller is extended to fetch all workspace projects via the batched GraphQL query. The
client already handles SSE events and re-renders — no new client-side timer is added.

A settings section in the dashboard allows the user to click "+" to open an HTML
`<dialog>` with a single `<input>` for the repo slug. On submit, the server calls
`repository.projectsV2`, discovers the project, writes workspace.json, and emits an SSE
event. The new tab appears without a page reload. Each project tab has an "×" to remove
it (`DELETE /api/workspace/remove`), also followed by an SSE event.

### /init integration

Running `/init` in a new repo includes the step: *"Register this project in your workspace
dashboard?"*. Opting in writes the entry to workspace.json. Post-init, the developer can
manage the workspace list with:

```
/init workspace --list
/init workspace --add owner/repo
/init workspace --remove owner/repo
```

These sub-commands are dispatched in Phase 1 of `/init` — if `$ARGUMENTS` starts with
`workspace`, the normal `/init` phases are skipped and only the sub-command handler runs.

### Triage

`roxabi triage set #42 --status "In Progress"` auto-discovers the project linked to the
current repo via `repository.projectsV2` (one extra round-trip, result cached in-process).
The discovered `projectId` is cached for the duration of the process. `.env` values
override auto-discovery when present.

### Backward compatibility

A repo with only a `.env` file (no workspace.json) continues to work exactly as today.
`resolveConfig()` falls through to `shared/config.ts` when workspace.json is absent or
contains no entry for the current repo.

## Data Structures

### workspace.json

```json
{
  "projects": [
    { "repo": "Roxabi/roxabi-plugins", "projectId": "PVT_kwDORa9q-M4Aqkwn", "label": "Roxabi Plugins" },
    { "repo": "mickaelV0/repo-b",      "projectId": "PVT_...",              "label": "Personal Repo B" }
  ]
}
```

Fields: `repo` (owner/name), `projectId` (GitHub PVT_ node ID), `label` (display name in dashboard tab).

### CLI package structure

```
cli/
├── package.json          # name: "roxabi-cli", version: "0.1.0", bin: { roxabi: "./index.ts" }
├── index.ts              # subcommand router: workspace | issues | triage | dashboard
├── lib/
│   └── workspace.ts      # readWorkspace(), writeWorkspace(), discoverProject()
└── commands/
    ├── workspace.ts      # list / add / remove
    ├── issues.ts         # fetch all projects → combined table
    ├── triage.ts         # set / list / create
    └── dashboard.ts      # start / stop / status
```

## Breadboard

### CLI surfaces

| Affordance | Handler | Data in | Data out |
|------------|---------|---------|----------|
| `roxabi workspace list` | `commands/workspace.ts` | workspace.json | Table: repo, projectId, label |
| `roxabi workspace add <repo>` | `commands/workspace.ts` | repo slug | GraphQL discovery → prompt if multiple → workspace.json |
| `roxabi workspace remove <repo>` | `commands/workspace.ts` | repo slug | workspace.json updated; exit 1 + error if not found |
| `roxabi issues [--project PVT_x]` | `commands/issues.ts` | workspace.json | Batched GraphQL (1 request) → combined terminal table |
| `roxabi triage set #N --status X` | `commands/triage.ts` | issue#, status | GraphQL auto-discovery → update mutation |
| `roxabi dashboard start/stop/status` | `commands/dashboard.ts` | — | Process control |

### Dashboard HTTP API (local server)

| Endpoint | Method | Handler | Effect |
|----------|--------|---------|--------|
| `/api/events` | SSE | existing | Push `data-refresh` event on workspace change or poll tick |
| `/api/workspace/add` | POST `{repo}` | `server/workspace.ts` | Discover project → write workspace.json → emit SSE → 200 |
| `/api/workspace/remove` | DELETE `{repo}` | `server/workspace.ts` | Remove from workspace.json → emit SSE → 200 |

### Dashboard browser (HTML)

| Affordance | Handler | Behaviour |
|------------|---------|-----------|
| Tab bar `[All]` click | `page.ts` | Show grouped All view |
| Tab bar `[Repo X]` click | `page.ts` | Show per-project view for Repo X |
| "+" button | `page.ts` → `<dialog>` → POST `/api/workspace/add` | Open `<dialog>` with `<input>` → submit → tab appears via SSE re-render |
| "×" on tab | `page.ts` → DELETE `/api/workspace/remove` | Remove → SSE re-render; show empty state if last project removed |
| SSE `data-refresh` event | `page.ts` | Re-fetch and re-render with fresh data + update "Last updated" timestamp |

### /init workspace sub-commands

| Affordance | Handler | Data in | Data out |
|------------|---------|---------|----------|
| `/init workspace --list` | Phase 1 dispatch in `init.ts` | workspace.json | Table: repo, projectId, label (same columns as `roxabi workspace list`) |
| `/init workspace --add owner/repo` | Phase 1 dispatch | repo slug | GraphQL discovery → workspace.json |
| `/init workspace --remove owner/repo` | Phase 1 dispatch | repo slug | workspace.json updated; exit 1 + error if not found |

### Config resolution (`issues/lib/config.ts`)

| Source | Priority | Condition |
|--------|----------|-----------|
| `lib/workspace.ts` → workspace.json entry for current repo | 1 (highest) | workspace.json exists + has matching `repo` entry |
| `shared/config.ts` → `process.env` | 2 (fallback) | Env vars `PROJECT_ID`, `STATUS_FIELD_ID`, etc. |

## Success Criteria

### CLI installation
- [ ] `bun install -g roxabi-cli` succeeds; `roxabi --version` prints a semver string
- [ ] `roxabi --help` lists all subcommands (`workspace`, `issues`, `triage`, `dashboard`)

### Workspace management
- [ ] `roxabi workspace add Roxabi/roxabi-plugins` — when exactly one GitHub Project is linked — discovers it via GraphQL and writes to workspace.json without requiring any env vars
- [ ] `roxabi workspace add Roxabi/roxabi-plugins` — when multiple projects are linked — prints a numbered list and writes the user-selected entry
- [ ] `roxabi workspace list` prints a table with columns `repo`, `projectId`, `label` for all registered projects
- [ ] `roxabi workspace remove Roxabi/roxabi-plugins` removes the entry; `roxabi workspace list` no longer shows it; exits 0 with a confirmation message
- [ ] `roxabi workspace remove unknown/repo` exits 1 with an actionable error message
- [ ] On a machine where neither `~/.roxabi-vault/` nor `~/.config/roxabi/` exist, `roxabi workspace add` creates `~/.config/roxabi/workspace.json` with parent directory mode `0700` and succeeds
- [ ] workspace.json is written to `~/.roxabi-vault/workspace.json` when vault exists, and to `~/.config/roxabi/workspace.json` otherwise

### Multi-project issues
- [ ] A unit test in `cli/__tests__/issues.test.ts` mocks `fetch` and asserts exactly **one** HTTP call is made when workspace.json lists two projects
- [ ] Each issue row in `roxabi issues` output includes the source project label column

### /init integration
- [ ] Running `/init` on a new repo presents the prompt "Register this project in your workspace dashboard?" and writes to workspace.json when the user opts in
- [ ] `/init workspace --list` outputs a table with the same columns (`repo`, `projectId`, `label`) and in the same order as `roxabi workspace list`
- [ ] `/init workspace --add owner/repo` adds an entry without executing any other `/init` phase
- [ ] `/init workspace --remove owner/repo` removes an entry; exits 1 with an error message if the repo was not registered

### Dashboard
- [ ] Dashboard tab bar renders `[ All ]` plus one tab per project in workspace.json
- [ ] All tab shows issues grouped by project with a `## {label}` section header per project
- [ ] Clicking a project tab shows only that project's issues (identical to today's single-project view)
- [ ] Dashboard "+" button opens an HTML `<dialog>` with a text input for repo slug; on submit a new tab appears without a full page reload
- [ ] Dashboard "×" on a tab removes the project from workspace.json and the tab disappears without a full page reload
- [ ] Removing the last project tab shows an empty-state message "No projects registered — click + to add one" on the All tab
- [ ] Dashboard shows a "Last updated: HH:MM:SS" timestamp that updates on every SSE-triggered refresh
- [ ] If `/api/workspace/add` or the SSE stream returns a non-200 status, the dashboard retains the last known data and the timestamp changes to "Last updated: HH:MM:SS (stale)"

### Backward compatibility
- [ ] A repo with only `.env` vars (`PROJECT_ID`, `STATUS_FIELD_ID`, etc.) and no workspace.json entry continues to work with `/issues` and `/issue-triage` exactly as before
- [ ] `roxabi triage set` in a repo with a `.env` file and no workspace.json entry uses `.env` values and succeeds

### Triage
- [ ] `roxabi triage set #2 --status "In Progress"` succeeds in the `roxabi-plugins` repo without any `.env` file, using auto-discovered project
- [ ] Running `roxabi triage set` outside a git repository prints an actionable error message and exits 1
