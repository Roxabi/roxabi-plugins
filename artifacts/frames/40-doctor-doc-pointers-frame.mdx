## Problem

`/doctor` verifies dev-core config health (stack.yml fields, GitHub project, labels, branch protection) but has no awareness of the documentation structure it helped set up. After `/init` runs Phase 7 (scaffold-docs) or Phase 7b (scaffold-fumadocs), there is no subsequent validation that `docs.path` actually exists, that key standard docs are present, or that `standards.*` paths in `stack.yml` point to real files. Developers who skip Phase 7 or whose doc files get deleted have no feedback loop.

## Who

- **Primary:** Developers running `/doctor` on a dev-core project who want a full health picture — including whether their documentation structure is set up and referenced correctly in `stack.yml`.
- **Secondary:** dev-core agents (backend-dev, frontend-dev, tester, fixer) that read `standards.*` paths from `stack.yml` — they silently get nothing if those paths are missing.

## Constraints

- Doctor must remain non-destructive: check-only by default, interactive repair on request.
- "Pattern docs" means the MDX files scaffolded by #38: `docs/architecture/index.mdx`, `docs/standards/backend-patterns.mdx`, `docs/standards/testing.mdx`, etc. — not ARCHITECTURE.md / PATTERNS.md (pre-#38 naming).
- "Import/refresh" AC is now superseded by #38: doctor should offer to run `scaffold-docs` or `scaffold-fumadocs` (via init.ts) rather than importing from Lyra or roxabi-plugins templates.
- Doctor output is a summary table — docs status must slot into the existing table format cleanly.

## Out of Scope

- Importing docs from Lyra or external repos (superseded by scaffold-docs in #38).
- Checking doc content quality — only presence and path resolution.
- Creating or modifying doc files directly (delegate to scaffold-docs / scaffold-fumadocs).
- Changes to any skill other than `/doctor`.

## Complexity

**Tier: F-lite** — Single domain (doctor SKILL.md + doctor.ts), clear scope, no new architectural patterns. Four checks to add to an existing phase, one new interactive repair offer.

Signals observed:
- Touches 1–2 files (SKILL.md, possibly doctor.ts for path-check logic)
- No new abstractions — extends existing Phase 2 health-check pattern
- All 5 ACs are concrete and bounded
