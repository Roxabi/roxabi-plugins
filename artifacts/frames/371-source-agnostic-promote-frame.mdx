## Problem

`/promote`'s version **derivation** is already source-agnostic. #353 rebuilt it on tag reachability: `price.sh` takes `COMPONENT ANCHOR HEAD` and computes the next `<component>/vX.Y.Z` from what is reachable on `main` + the conventional-commit deltas; `finalize.ts` is a pure classifier taking `isPromote` as a plain bool. Neither reads the string `staging`.

The `staging â†’ main` release-train lives entirely in the **policy** layer wrapped around that derivation, and it is thin â€” three sites:

- `release-consistency.yml` PR path: early-greens unless `PR_HEAD_REF == staging` (~1-line branch gate).
- `preflight.sh`: hardcodes `git checkout staging` + `git log main..staging`.
- promote `SKILL.md` step 9b: derives `isPromote` from "newest merged base=main **head=staging** PR", plus the three-way **witness** check (the D7 witness machinery introduced by #353).

This coupling contradicts dev-core's own Design Principle #1 (project-agnostic) and blocks any consuming repo from going trunk-based (`main`-only) without forking the promote path â€” which would mean roxabi-plugins ships dev-core tooling it no longer dogfoods.

**Why now:** roxabi-plugins wants to make `main` the default and delete `staging` â€” that plan is blocked on this. The operator has already gone version-less on his other plugins and it works fine; Model B gives that same zero-per-PR-discipline ergonomics but with **real semver `<component>/v*` tags** instead of SHA-only versions.

## Who

- **Primary:** the operator/maintainer of any repo that wants trunk-based flow. Today `/promote` assumes a stagingâ†’main train and the `release-consistency` gate never fires on a `main`-only repo, so there is no supported path.
- **Secondary:** the whole dev-core fleet currently on the staging train (`roxabi-factory`, `roxabi-live`, `roxabi-boilerplate`) â€” they must keep releasing unchanged until they opt in. Plus every downstream tag consumer resolving `<component>/vX.Y.Z` (`[tool.uv.sources]` pin-swap, `on: push: tags` build triggers in `voiceCLI`/`llmCLI`/`imageCLI`).

## Constraints

- Tag convention A is fixed: `<component>/vX.Y.Z`. The derivation (`price.sh`) is correct and stays â€” reuse as-is, do not rebuild.
- **Backward-compatible via a transition flag.** Existing staging-train repos keep current behavior by default; `main`-only is opt-in. No flag-day fleet migration.
- `release:` config lives in the **existing** `.claude/stack.yml` (per #353's constraint), not a new file.
- Amends #353 decisions **D8 / D9 / D14** (witness three-way + `head=staging` policy) â€” must **not** regress #353's structural win: one writer, one value computed once and projected, version-fileâ†”tag drift structurally impossible.
- Dogfood on roxabi-plugins' own migration before recommending fleet adoption.
- Reconcile the two divergent version surfaces: `plugins/dev-core/.claude-plugin/plugin.json` `version` (marketplace install, bumped by the `check-skill-version` pre-push hook â€” now `0.8.24`) vs `dev-core/v*` git tags (promote).

## Out of Scope

- Actually executing roxabi-plugins' `main`-default + delete-`staging` migration â€” that is the **unblocked follow-up**, a separate issue, not this one.
- Migrating `roxabi-factory` / `roxabi-live` / `roxabi-boilerplate` off the staging train â€” opt-in, per-repo, later.
- **Model A** (parameterised `release.source_branch` that keeps the witness machinery) â€” evaluated and rejected in the decision; not re-litigated here.
- Multi-component repos (`roxabi-factory`) â€” `/promote` is single-component by construction (same boundary #353 drew).

## Premise Validity

**Success in 6 months:** roxabi-plugins runs `main`-only and gets correct `<component>/vX.Y.Z` tags + GitHub releases created **automatically on merge-to-main**, using unchanged fleet tooling. Staging-train repos (`factory`, `live`, `boilerplate`) still release green with **zero** changes, via the transition-flag default. A repo switches modes by flipping one `release.model` value, nothing else.

**Failure in 6 months (falsifiable):** either (a) `adopted_repos(main-only) == 0` six months after ship â€” the flag exists but nobody migrated, so the staging coupling was **not** actually the blocker (premise invalid); OR (b) `â‰Ą1` staging-train repo had a broken or missed release attributable to this change and had to pin an older dev-core version. Both are observable counts/booleans against a fixed horizon.

**Simplest alternative:** patch only roxabi-plugins â€” locally override `release-consistency`'s branch gate + `preflight.sh` so this one repo goes `main`-only, and leave dev-core's staging coupling intact.

**Why not simplest:** that *is* the divergence this work exists to remove. roxabi-plugins would ship a promote path it forks locally; every other repo that later wants `main`-only re-solves it from scratch; the shared tool stays project-coupled â€” the exact opposite of "source-agnostic in the shared tool." #353's own lesson applies: two mechanisms with no reconciler is the drift generator itself.
