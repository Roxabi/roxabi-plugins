## Problem

Most dev-core agents (55–70 lines) are thin wrappers that delegate all domain knowledge to `{standards.xxx}` references from `stack.yml`. Only `security-auditor.md` (172 lines) embeds actionable checklists, severity matrices, and concrete patterns.

This creates a fragile dependency: if `stack.yml` is missing or its standards paths point to sparse/nonexistent docs, agents produce generic, low-value output. The gap between `security-auditor` and every other agent is stark — 172 lines of embedded expertise vs 55–70 lines of delegation.

## Who

- **Primary:** dev-core plugin users — teams relying on agents for code review, implementation, and architecture decisions
- **Secondary:** plugin maintainers — enriched agents are more self-contained and easier to test/validate

## Constraints

- Each agent must remain under 250 lines (compressed notation preferred)
- Existing behavior (escalation thresholds, communication boundaries, Phase 0 guards) must not change
- Agents must still read `{standards.xxx}` when available — embedded knowledge is the fallback, not a replacement
- Use `security-auditor.md` as the template: structured checklists, severity tables, concrete examples

## Out of Scope

- Changing agent logic, tool permissions, or escalation flows
- Refactoring agent file structure or naming
- Adding new agents
- Modifying `security-auditor.md` (it's already the gold standard)

## Complexity

**Tier: F-lite** — Clear scope (documentation enrichment), single domain (agent definitions), no architectural changes.
