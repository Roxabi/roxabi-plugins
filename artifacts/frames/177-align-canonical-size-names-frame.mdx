## Problem

`dev-core:issue-triage` documents canonical size names **S / F-lite / F-full**, but several active Roxabi boards (e.g. `Roxabi/lyra`) still carry the legacy GitHub Projects v2 size options **XS / S / M / L / XL**. When `dev-core.yml.size_options_json` is generated from a legacy board, it only holds the legacy keys.

Result: passing a canonical name on a legacy board fails. `resolveSize('F-full')` returns the canonical `'F-full'` (it's a valid `CANONICAL_SIZES` member), but `SIZE_OPTIONS['F-full']` is `undefined` on the board → `create.ts` / `set.ts` size-on-board path abort with `Error: Invalid size`. The skill's documented canonical names are non-functional on any unmigrated board.

`a29b5e6` (2026-05-26) reshaped `resolveSize` to check the project's `SIZE_OPTIONS` first — but that fixed the *reverse* direction (legacy name on legacy board, e.g. `XS` no longer folds to `S`). The canonical-name-on-legacy-board path this issue describes **still reproduces**.

Secondary: the priority label sync in `set.ts`/`create.ts` issues `gh issue edit ... --remove-label P0-critical,P1-high,P3-low` to clear sibling priority labels. If any sibling doesn't exist in the target repo, the whole sync aborts (`'P0-critical' not found`).

## Who

- **Primary:** anyone running `/dev`, `/issue-triage`, or `triage.ts create/set --size <canonical>` against a Roxabi board still on the legacy XS/S/M/L/XL schema (currently `lyra`, others unmigrated).
- **Secondary:** maintainers carrying per-board `dev-core.yml` stopgap alias keys; future boards generated by `/init` from legacy templates.

## Constraints

- Single domain: `dev-core/skills/issue-triage` TS lib (`lib/create.ts`, `lib/set.ts`) + `shared/adapters/config-helpers.ts`.
- Must not regress `a29b5e6` (legacy-name-on-legacy-board preservation) or the canonical-on-canonical-board path.
- Zero board mutation in this cycle — runtime fallback only.
- Bun + vitest; biome formatting; semver-disciplined plugin → version bump on release.

## Out of Scope

- **Option A** (`migrate rename-size-options` GraphQL command that renames board options) — separable board-ops tool; own issue.
- Priority *option* canonicalization (current canonicals already match legacy boards).
- Lane field (project-specific, tolerates board-defined names).

## Premise Validity

**Success in 6 months:** on any legacy-schema board, `triage create/set --size F-full` (canonical) succeeds and sets the correct board option; zero `Invalid size` errors across Roxabi boards; lyra's `dev-core.yml` stopgap alias keys can be removed.

**Failure in 6 months:** ≥1 `Invalid size` error still occurs on a legacy board when a canonical size is passed after this ships, OR the manual `dev-core.yml` alias-key stopgap remains required beyond 3 months.

**Simplest alternative:** keep the per-board `dev-core.yml` alias-key stopgap (manually map `"F-lite":"<M-id>","F-full":"<XL-id>"`).
**Why not simplest:** doesn't scale — every legacy board needs hand-patching, the patch is wiped on `size_options_json` regen, and the documented canonical names stay non-functional by default.
