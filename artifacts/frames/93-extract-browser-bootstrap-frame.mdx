## Problem

Three plugins currently hand-roll the same `playwright` + `playwright-stealth` bootstrap with subtle divergence:

| Plugin | File | Variant | Context lifecycle | Purpose |
|---|---|---|---|---|
| `web-intel` | `scripts/fetchers/stealth.py` | sync | ephemeral | anti-bot fallback for `/scrape` |
| `web-intel` | `scripts/screenshot.py` | sync | ephemeral | full-page PNG for `/roast`, `/benchmark` |
| `linkedin-apply` | `scripts/scraper.py` (`get_browser_context`) | async | `launch_persistent_context` | LinkedIn session reuse |

Each copy is ~30–40 LoC of import guards, stealth instantiation, UA/viewport config, and `webdriver-off` args. The copies drift:

- Bug fixes land on one side only — the error-surfacing refactor across commits `80387a7..7f58dbd` updated web-intel but not linkedin-apply.
- Sync and async variants duplicate the same stealth setup twice.
- New web-intel commits (`fa46b9b docs: annotate duplicated playwright+stealth bootstrap across plugins`) explicitly flag this as deferred tech debt.

Observable impact: next time a stealth flag needs tuning or an import guard needs tightening, the fix has to be triple-applied or one of the consumers silently regresses.

## Who

- **Primary:** Mickael, as maintainer of `web-intel` and `linkedin-apply` — fixes and feature work on either plugin currently need mirroring.
- **Secondary:** Future plugins that need a stealth browser (e.g. `agent-browser`, any new scraper) — today they'd copy one of the three variants instead of importing a shared helper.

## Constraints

- **Preserve the `(html, error)` tuple contract** in `web-intel/scripts/fetchers/stealth.py::fetch_html_stealth()` — the error-surfacing refactor from `7f58dbd` must survive.
- **Preserve LinkedIn auth semantics** — `linkedin-apply` relies on `launch_persistent_context` with a cookie/profile directory. Any cookie-jar or storage-state regression breaks the live LinkedIn session.
- **Single new module** — `roxabi_sdk/browser.py`, synced via `tools/sync_sdk.sh` into each plugin cache like the rest of `roxabi_sdk/`.
- **Typed error on missing dep** — callers currently bail with generic `ImportError`; the new helper must raise a typed `PlaywrightNotAvailableError` (or similar) with install instructions.
- **Caller owns lifecycle** — return `(Playwright, BrowserContext, Page)` so call sites keep their existing close/cleanup structure.
- **Both sync and async entry points** — web-intel is sync, linkedin-apply is async; one module, two functions.

## Out of Scope

- **Rewriting the call sites beyond the bootstrap swap** — `fetch_html_stealth` retry logic, screenshot capture pipeline, and LinkedIn scraper selectors are untouched.
- **Adding new stealth flags or anti-detection tricks** — parity only; behavior change is a non-goal.
- **Migrating `agent-browser` or any other plugin** — only the three call sites listed in the issue.
- **Switching drivers** — no Selenium/CDP alternatives; Playwright stays.
- **Changing persistent-context location or format** — same `user_data_dir` path linkedin-apply already uses.
- **Adding retry/circuit-breaker in the helper** — callers own resilience.

## Complexity

**Tier: F-full** — Multi-domain refactor spanning the shared SDK plus two plugins with divergent runtime models (sync web-intel vs async linkedin-apply, ephemeral vs persistent context). The API shape isn't obvious up front — the analyze step exists specifically to reconcile sync/async signatures, lifecycle ownership, and the typed-error contract before spec.

Signals observed:
- Cross-plugin: `roxabi_sdk` + `web-intel` + `linkedin-apply` (3 surfaces)
- Two runtime models to unify (sync/async, ephemeral/persistent)
- Auth-sensitive risk vector (LinkedIn persistent session)
- Behavior contracts to preserve (`(html, error)` tuple, cookie jar semantics)
- 6 concrete touchpoints: 1 new module, 3 call-site refactors, 1 test file, 1 doctor update
