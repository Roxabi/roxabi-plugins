## Problem

Every Roxabi Python project should enforce the same code-hygiene guards that `lyra` already has — file-length cap (`check_file_length.sh`), folder-size cap (`check_folder_size.sh`), and import-layer contracts (`import-linter`). Today these guards are hand-copied per repo, so scaffolding a new project via `/init` leaves them absent and the drift is noticed late (e.g. when `voiceCLI/cli.py` grows to 1572 LOC and nobody is warned).

Audit as of 2026-04-18 shows only `lyra` and the freshly-ported `llmCLI` have the full set. `voiceCLI` has 0 of 3 guards and 7 files over 300 LOC (max: `cli.py` 1572, `api.py` 1107, `stt_daemon.py` 993). `imageCLI` also has 0 of 3 and 7 files over 300 LOC (max: `cli.py` 807). `voiceCLI` also has one folder (`src/voicecli`, 19 files) over the 12-file budget.

Root cause: no declarative contract in `stack.yml` that tells `/init` and `/release-setup` to install these guards. Adding one makes it enforceable, idempotent, and auditable across every Roxabi Python repo.

## Who

- **Primary:** Maintainer (Mickael) scaffolding or updating Roxabi Python repos — `roxabi-plugins/dev-core` is the source of truth for project setup
- **Secondary:** Future contributors / agents who land in a repo and expect the same guardrails everywhere (no surprise that one repo rejects a 500 LOC file and another does not)

## Constraints

- **Canonical source of truth** — the two shell scripts must live in `plugins/dev-core/tools/` (per decision), not be re-duplicated per project. Project-side copies are stamped from here on `/init` or `/release-setup --force`
- **Idempotent** — `/release-setup --force` must be runnable on a repo with partial guards (e.g. `lyra` already has them; only missing `stack.yml` section) without clobbering or duplicating hooks
- **Per-project exemption files** — `tools/file_exemptions.txt` and `tools/folder_exemptions.txt` stay in each project (not in dev-core), seeded empty with header comments on install
- **Project-appropriate import layers** — `.importlinter` scaffold must be generic enough to land in any project; layer names (cli/mid/base or equivalent) either auto-detected from `src/` structure or templated with clear TODOs
- **Hook ordering** — the three pre-commit hooks (`check-file-length`, `check-folder-size`, `import-layers`) go after `typecheck`, before `gitleaks` (matches lyra's order)
- **Backwards-compat** — projects without `quality_gates:` section in `stack.yml` keep working unchanged; no auto-install, opt-in only
- **Cross-repo refactors** — splitting `cli.py` (1572 → multiple files), `api.py` (1107 → multiple files), `stt_daemon.py` (993 → multiple files) in `voiceCLI` must be behavior-preserving; same for `imageCLI/cli.py` (807) and its 500+ LOC engine files

## Out of Scope

- Non-Python repos (TypeScript/Bun projects in roxabi-plugins itself) — this issue is Python-specific
- New guard types (line-length, cyclomatic complexity, doc-coverage, etc) — the three guards listed are the whole set
- Changing the 300 / 12 / layer contract thresholds — they stay at the lyra defaults; per-project override via `stack.yml` only
- Retroactively tagging exempt files with tracking issues — exemptions get seeded with a TODO comment pointing to the tracking issue once the split issue is filed, but issue-filing itself is part of the per-repo follow-up cycle
- Migrating lyra's scripts into dev-core — the scripts are already in lyra and llmCLI; this issue adds `plugins/dev-core/tools/` as the new canonical source, with lyra/llmCLI re-stamped from it on their next `/release-setup --force`
