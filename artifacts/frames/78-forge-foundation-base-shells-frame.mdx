## Problem

Forge's design system lives in markdown prose (tokens.md ~2000 lines, split-file.md). Every invocation forces Claude to re-read prose, re-derive CSS, and regenerate from scratch — causing visual drift across outputs and token waste. Advanced components (phase badges, lane containers, dependency paths) had to be hand-built in nats-arch-roadmap.html because forge lacked embeddable classes.

The gallery-templates system (gallery-base.css/js) already proves the actual-file pattern works: Claude reads once, embeds directly, no prose re-derivation. base/ + shells/ extends this pattern to all forge output.

## Who

- **Primary:** Mickael — produces forge diagrams/guides across lyra, roxabi-plugins, voiceCLI, 2ndBrain
- **Secondary:** downstream issues #79–#83 — aesthetics, components, and skill rewrites all depend on base/ existing

## Constraints

- ADR-011 decisions: theme-toggle.js stays inline in shells (15 lines); semantic color tokens live in components.css; anti-patterns as CSS comments (structural)
- All CSS must use custom properties — no hardcoded colors
- Gallery system (gallery-base.css/js) must not be affected — separate files, separate paths
- Must work with `file://` protocol (self-contained, no external fetches)
- Shell placeholders must include explicit instructions for each slot

## Out of Scope

- Aesthetic CSS files (lyra.css, roxabi.css, etc.) — #79, #80, #81
- Component library (.phase, .lane, .card.severity, etc.) — #82
- Skill rewrites (forge-chart, forge-guide SKILL.md updates) — #83
- tokens.md / split-file.md slimming — happens after skills point to new files
