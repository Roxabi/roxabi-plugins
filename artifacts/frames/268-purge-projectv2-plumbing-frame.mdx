## Problem

#267 dropped the ProjectV2 board and relocated issue management to the `roxabi-issues`
plugin (issues-only / labels-first model). It removed the board **skills** and most
**scaffolding**, but deliberately left the underlying ProjectV2 **plumbing** in place to
keep that decouple green. That residue is now split across two plugins and behaves in two
different ways:

- **dev-core shared layer** — `config-helpers.ts` (32 ProjectV2/`*_OPTIONS` refs),
  `env-config.ts` (`GH_PROJECT_ID` / `*_FIELD_ID` loading), `queries.ts` (33 ProjectV2
  GraphQL refs), `github-adapter.ts` (`addProjectV2ItemById` …). **Orphaned dead code** —
  grep finds zero importers in dev-core skills. Harmless to the build, but a trap.
- **dev-init `init scaffold`** — `init/lib/scaffold.ts` (14 board refs) still accepts
  `--project-id` / `--status-field-id` / `--size-field-id` / `--priority-field-id` and
  **writes them** into `.claude/dev-core.yml` / `.env`. This path is **live**: `/init` still
  bootstraps a board that no longer exists in the issues-only model → misleading UX.
- **dev-init shared/** — a `@generated` mirror of the dev-core canonical (`queries.ts`
  ProjectV2 count 33/33 identical). Auto-covered: regenerated by `bun run sync:shared`
  once the canonical is purged.

Leaving this half-done makes #267's decouple permanently incomplete and risks the board
patterns being copied or re-wired by a future maintainer.

## Who

- **Primary:** maintainers of dev-core / dev-init — every `/init`, `/dev`, `/pr` reads this
  shared layer; the live scaffold path actively mis-bootstraps new repos.
- **Secondary:** anyone running `/init` on a fresh project — gets board field prompts/IDs
  for a board model that was removed.

## Constraints

- Removal cascades through `env-config` + `scaffold` + their tests — this is why #267
  deferred it.
- `dev-init/skills/shared/*` is `@generated` (`DO NOT EDIT` banner) — never hand-edit;
  purge the dev-core canonical then run `bun run sync:shared`.
- `sync:shared --check` must stay green (drift gate).
- No behaviour change to the issues-only path (`roxabi-issues`) — this is pure subtraction.

## Out of Scope

- Any change to `roxabi-issues` plugin (already issues-only, in roxabi-live).
- Re-introducing a board read path (that is the cockpit's job, separate concern).
- Touching `artifacts/` historical frames/specs/plans that mention ProjectV2 (traceability).

## Premise Validity

**Success in 6 months:** `grep -rE 'ProjectV2|GH_PROJECT_ID|_FIELD_ID'` across
`plugins/dev-core` + `plugins/dev-init` source returns **0** matches; `init scaffold` no
longer accepts or writes any board field; typecheck + lint + tests + `sync:shared --check`
green. The board model is gone, not dormant.

**Failure in 6 months (falsifiable):** the grep above still returns **> 0** matches in
either plugin's source, OR `/init` still writes `GH_PROJECT_ID` to `dev-core.yml` — i.e.
the cleanup was closed but the plumbing survived. Directly observable, decidable now.

**Simplest alternative:** leave the dead code — dev-core's copy is orphaned and the build
is already green.
**Why not simplest:** it does not address dev-init's `init scaffold`, which is **live** and
still mis-bootstraps a non-existent board; and the orphaned dev-core code is a re-wire trap
(field maps + GraphQL ready to copy). Half-measure = #267 stays permanently incomplete.

## Complexity

**Tier: F-full** — single domain but real architectural impact: trims two hexagonal port
contracts (`ConfigPort`, and retires `IssuePort`/`ProjectPort`), removes a DI layer, converges
a `intentional-fork` file back to copy-sync, and reconciles ADR-005/006 via ADR-015. ~20 source
files across 2 plugins + a sizable test surface + docs + manifest/CLAUDE.md governance update.

Signals observed:
- Architectural impact: port-contract changes + ADR supersession (not pure subtraction).
- 15+ files: dev-core shared layer (config-helpers, env-config, ConfigPort, queries, types,
  ports/workspace, workspace-store, github-adapter) + dev-init (init, scaffold, **delete**
  ports/issue, ports/project, services, GitHubAdapter class) + ~6 test files + checkup/stack-setup
  docs + `tools/shared-sources.json` + CLAUDE.md.
- Cross-file coupling through mixed modules + a copy-sync convergence step.
- Mostly **deletion** (low risk) but broad blast radius → F-full process, not F-lite.
