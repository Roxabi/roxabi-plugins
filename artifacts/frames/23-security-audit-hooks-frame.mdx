## Problem

The roxabi-plugins dev-core skills perform high-risk git operations (push, merge, PR creation) but there is no systematic audit confirming they follow Claude Code security best practices. Three knowledge resources (`32be5f8e`, `4e56c7f3`, `47a21af1`) document hook patterns, permission isolation, and review gates that may not be fully implemented.

The risk is that an agent running `/implement`, `/pr`, or `/promote` could accidentally force-push, skip hooks (`--no-verify`), or merge to the wrong branch — actions that are hard to reverse and affect shared state.

## Who

- **Primary:** Developers using dev-core skills in production repos — they trust the skills to be safe by default.
- **Secondary:** Repo maintainers reviewing PRs generated by agents — they need confidence that agent-generated changes follow guardrails.

## Constraints

- No special constraints. Standard dev workflow applies.
- Three knowledge resources to audit against (IDs: `32be5f8e`, `4e56c7f3`, `47a21af1`).
- Related: issue #19 (dev-core agent reasoning & context audit).

## Out of Scope

- Nothing excluded — all 6 tasks from the issue body are in scope:
  1. Audit hooks compliance across skills
  2. Implement anti-accidental-merge hook pattern
  3. Verify minimal permissions in SKILL.md frontmatter
  4. Align SKILL.md template with 15 best practices
  5. Document in CONTRIBUTING.md
  6. Security review of implement, pr, promote skills

## Complexity

**Tier: F-lite** — Single domain (security/hooks), clear scope from issue body, no architectural unknowns. Audit + targeted fixes across existing files.

- Single plugin (dev-core) as primary target
- Well-defined checklist from knowledge resources
- Changes are edits to existing skills/templates, not new architecture
