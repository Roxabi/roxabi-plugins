## Problem

The plugin codebase lacks foundational Clean Architecture and Hexagonal Architecture patterns needed for long-term maintainability. While high-level separation exists (skills vs scripts), internal boundaries are blurred: adapters import infrastructure directly, data access is coupled to SQLite, structured data travels as raw dicts, and business logic is scattered across CLI entrypoints.

Current hexagonal score: ~5/10. The absence of formal ports, repository abstractions, domain models, and a use-case layer makes every new plugin harder to build and every infrastructure change (e.g., storage migration) a repo-wide effort.

The `_lib/paths.py` file is vendored 5 times — a concrete symptom of missing shared infrastructure.

## Who

- **Primary:** Plugin developers (Roxabi team) — every new plugin or feature touches these pain points.
- **Secondary:** Contributors and consumers of individual plugins (vault, web-intel, cv, etc.) — they inherit fragile error handling, untyped data, and tight coupling.

## Constraints

- No special constraints. Standard incremental migration — don't break existing functionality.
- All existing tests must continue to pass; coverage must not regress.
- Each plugin is self-contained, so plugins can be migrated independently.

## Out of Scope

- Database migration (SQLite to another store) — this refactor creates the abstraction layer; actual migration is a separate effort.
- New plugin features — this is purely structural refactoring.
- CI/CD pipeline changes beyond what's needed for the SDK package.

## Scope

Both **Python plugins** (web-intel, vault, cv, get-invoice-details, linkedin-apply, linkedin-post-generator) and **TypeScript code** (dev-core) are in scope for Clean Architecture patterns.

### Python plugins — 8 pain points addressed

1. No formal Ports (interfaces) — `BaseFetcher` imports infrastructure directly
2. Hardcoded adapter routing — `web-intel/scraper.py` if/elif chain violates OCP
3. No Repository Pattern — plugins depend on `VaultDB` directly
4. No Domain Model — raw dicts everywhere, zero type safety
5. No Use Case layer — business logic in CLI entrypoints
6. No config schema validation — raw dict configs, silent failures
7. Duplicated infrastructure — `paths.py` vendored 5 times
8. Generic exceptions — `Exception` everywhere

### TypeScript (dev-core) — same patterns applied

- Formal port interfaces for GitHub API access, config loading, workspace management
- Domain models for issues, plans, specs (currently ad-hoc types)
- Repository pattern for artifact storage
- Use-case layer separating skill orchestration from implementation

### Migration phases (from issue)

| Phase | Scope | Breaking? |
|-------|-------|-----------|
| 1. Foundation | Port interfaces, domain models, exception hierarchy | No |
| 2. Adapter layer | Plugin-by-plugin: vault, web-intel, cv, invoice, linkedin | No |
| 3. Use case layer | Extract business logic from CLI entrypoints | No |
| 4. Plugin SDK | Shared `roxabi-sdk` package, remove vendored `paths.py` | No (internal) |

## Complexity

**Tier: F-full** — Multiple domains (6 Python plugins + 1 TS plugin), new architectural patterns (ports, repositories, use cases, SDK), complexity score 9/10. Requires deep analysis of each plugin's boundaries before implementation.
