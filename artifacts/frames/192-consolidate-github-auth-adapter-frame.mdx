## Problem

GitHub access logic in `dev-core` is reimplemented up to 5√ó. The dev-core audit flagged this as the biggest dedup ROI. Four distinct duplications:

- **D1** ‚ÄĒ token resolution (`env ‚Üí gh auth token`) duplicated √ó5: `github-adapter.ts:73`, `github-adapter.ts:359`, `cli/commands/issues.ts:15`, `cli/lib/github-discovery.ts:18`, `skills/issues/lib/fetch-workflow.ts:20`.
- **D2** ‚ÄĒ `GitHubAdapter` class vs standalone functions: ~10 methods duplicated (`getItemId`, `addToProject`, `updateField`, `getBoardIssueNumbers`, `link`‚Ä¶), `github-adapter.ts:73-101` vs `:358-500`. - **D3** ‚ÄĒ `ghGraphQL` (execSync + tmpfile) duplicated: `show.ts:27`, `digest.ts:33`.

Observable impact: every change to GitHub access touches multiple copies; the adapter keeps drifting; the class/standalone split is a stalled migration that doubles maintenance.

## Who

- **Primary:** dev-core maintainers ‚ÄĒ any change to GitHub access currently means editing 2‚Äď5 sites.
- **Secondary:** plugin consumers ‚ÄĒ fewer drift bugs, consistent auth behavior.

## Constraints

- Single domain: TypeScript GitHub-access layer under `plugins/dev-core`.
- Must preserve existing call-site behavior (no functional change ‚ÄĒ pure dedup/refactor).
- D2 carries the axial decision (keep standalone, drop class) ‚ÄĒ must respect the repo's axial ADR.
- No `--force`/`--hard`/`--amend`; biome formatting; single quotes, no semicolons.

## Out of Scope

- Touching `dev-init` copies of `github-adapter.ts` / `config-helpers.ts` (separate plugin).
- New GitHub features or API surface changes.
- Changing the GraphQL queries themselves (only their execution wrapper).

## Premise Validity

**Success in 6 months:** One canonical `getGitHubToken()`, one `GitHubAdapter` API, one `ghGraphQL`, and reuse of `detectGitHubRepo()`. Grep for each of the 5 dup patterns returns a single definition.

**Failure in 6 months:** Duplicate token/GraphQL/detectRepo implementations reappear (grep finds ‚Č•2 definitions of any), OR the class/standalone split persists with both still called from prod paths (count >1 of equivalent method sets).

**Simplest alternative:** Do only D1 (token dedup, highest ROI), leave D2‚ÄďD4.
**Why not simplest:** D2 is the N√ó2 wrong-axis duplication the issue centers on (~10 methods duplicated class-vs-standalone); leaving it means the adapter keeps drifting and the next change touches both copies. D3/D4 are cheap once D1 lands.
